Git Branching Guide

A plain-language visual guide for collaborating with Chris on your Google Meet app

01 What is "the remote"?

Think of your code as existing in two places. Your computer (called "local") is where you actually write and edit code. GitHub (called "the remote") is the shared copy in the cloud where you and Chris can both access the code.

Local (Your Computer)

Your personal copy of the code. You edit files here, make commits, and test things. Only you can see changes here until you push them.

Remote (GitHub)

The shared copy in the cloud. Both you and Chris can push code up to it and pull code down from it. Your remote is called work.

Analogy: Think of it like a shared Google Doc. Your local copy is like editing offline on your laptop. The remote is the live Google Doc that both of you sync with. "Pulling" downloads the latest version. "Pushing" uploads your changes.

02 The three types of branches

Branches are like parallel copies of your code that let you work on different things without interfering with each other.

main Production Branch

The "finished product." This is what Vercel deploys to your live website. Only tested, stable code goes here. Think of it as the published book.

dev Development Branch

The "work in progress" area. This is where you integrate and test new features before they're ready for production. Think of it as the draft manuscript.

feature/* Feature Branches

A temporary branch for one specific feature (like your feat/auto-extract-action-items). You create it, do your work, merge it back, and delete it. Think of it as writing one chapter on a separate piece of paper before adding it to the manuscript.

03 The ideal workflow

Here's how the branches should flow when everything is set up properly:

  main ──●──────────●───────────────────●──────────●──────►  (Vercel deploys here)
              ▲    │                         ▲
              │    │  branch off             │
              │    ▼                         │  merge when ready
              │   dev ──●────●────●────●────●──────────────────►
              │         ▲    │         ▲
    merge PR  │         │    │         │  merge when done
              │         │    ▼         │
              │         │   feat/A ──●──●──●
              │         │
              │         │
              │        feat/B ──●──●──●──●
              │
     (only after testing on dev)
Step by step
  1. Start a new feature by branching off dev (not main)
  2. Do all your work on the feature branch, committing as you go
  3. When the feature is done, merge it back into dev
  4. Test everything on dev to make sure nothing is broken
  5. When dev is stable, merge it into main — Vercel auto-deploys

What you should do now

You have two good options, and I'd recommend discussing with Chris which one he prefers:

Option A — Sync dev first (best practice):

  1. Merge main into dev so dev is caught up
  2. From now on, both of you branch features off dev
  3. Merge tested features from dev into main for deployment

Option B — Keep working off main (simpler for small team):

  1. Continue branching features off main
  2. Merge features directly back into main
  3. Use dev only for risky/experimental work (as Chris suggested)

Option B matches how Chris is already working and is totally fine for a two-person team. Option A is textbook best practice but adds overhead. Either way, the most important thing is that you and Chris agree on the same approach.

05 Common commands cheat sheet

Starting a new feature
# Make sure you have the latest code first
$ git pull work main

# Create a new branch for your feature
$ git checkout -b feat/my-new-feature

# Do your work, then save your changes
$ git add .
$ git commit -m "feat: describe what you did"

# Push your branch to GitHub
$ git push work feat/my-new-feature
Getting Chris's latest changes
# Download and merge the latest from main
$ git pull work main
Merging your feature when it's done
# Switch to main
$ git checkout main

# Get the latest
$ git pull work main

# Merge your feature in
$ git merge feat/my-new-feature

# Push to GitHub (triggers Vercel deploy!)
$ git push work main
If you want to sync dev with main
# Switch to dev
$ git checkout dev

# Merge all of main's changes into dev
$ git merge main

# Push the updated dev to GitHub
$ git push work dev
